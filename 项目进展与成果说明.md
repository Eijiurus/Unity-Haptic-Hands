# 项目进展与成果说明

**文档性质**：基于当前仓库代码、`README.md` 与 Git 提交历史整理的阶段性说明，用于汇报毕业设计进展与可展示成果。  
**工程定位**：Unity 2022.3 LTS 上的「基于数据手套的虚拟手交互与触觉反馈」原型系统（本科毕业设计方向）。  
**生成日期**：2026-05-11（v11 直接抓取 + v12 速率累加 + 触觉输出反馈链路落地版）  

---

## 一、项目目标（要解决什么问题）

在 Windows 上搭建一套可演示的闭环：**真实手指弯曲**驱动 **Rigged Hand 骨骼**，**手腕姿态与位移**来自 **BLE 惯性传感器**，场景中通过 **Trigger / 刚体** 完成触摸与抓取；触觉侧由 Unity 通过 UDP 把"播第几号 DRV2605 库振动"指令推送给 Python 桥（`HapticBridge.py`），Python 端再通过 USB HID 写入 DRV2605 双路 LRA 马达板——**结构与 SensorBridge.py 完全反向，形成"手套→Unity→振动板"的完整闭环**；根目录另保留 HID 离线测试脚本作为备用冒烟工具。

> 说明：`.cursorrules` 中曾提到 XR Interaction Toolkit；当前 `Packages/manifest.json` **未**引入 XRI，交互由自定义脚本与 Physics 完成，与主 `README.md` 描述一致。

## 三、成果展示（可按演示顺序对外说明）

### 1. 虚拟手与数据手套联动

- **成果**：五指弯曲实时映射到骨骼旋转；数据格式为手套 ASCII 行（分号结尾、11 个逗号分隔值，前 5 为弯曲通道），经 Python 去分号后以 UDP 整行送入 Unity。
- **对应实现**：`SensorBridge.py`、`GloveDataReceiver.cs`、`DataGloveHandDriver.cs`。

### 2. WitMotion 手腕跟踪

- **成果**：自动扫描名称含 **WT** 的 BLE 设备并连接；解析后用于手腕姿态，并支持更稳健的相对位置控制。
- **本轮进展**：`DataGloveHandDriver.cs` 已将位置部分从“依赖欧拉角补偿的自由积分”重构为 **慢速低通估计重力 + 启动静止标定 + 转动冻结窗口 + 弹簧回中** 的模型；并进一步把产品目标明确为**近场、短距离相对位移增强**（非远距离连续平移）：默认缩小 **`wristMaxOffset`**，可选 **`usePlanarPositionOnly`**（按轴向权重自动压掉最不稳定一轴，仅两轴参与位移）、**`positionRecenteringSpeed`**（运动过程中极慢拉回零以减轻长期累计），更适合答辩演示与近场抓取/碰触交互。
- **对应实现**：`WitMotionConnector.cs`、`Assets/Scripts/Bluetooth/` 下 `BleApi`、`BlueConnector`、`BlueScanner`，`Device/DeviceModel.cs`、`DevicesManager.cs`、`DataGloveHandDriver.cs`。

### 3. 场景内触摸与抓取

- **成果**：指尖球形 Trigger 检测可触摸物体；支持 **弯曲阈值捏取**、**拇指食指距离捏合**、**握拳抓取**、**G 键无手套捏取**；可选指尖物理碰撞球减轻穿模；可吸附刚体到指尖或手掌根。
- **对应实现**：`HandInteractionRig.cs`、`FingerTipRelay.cs`、`TouchableObject.cs`。

### 4. 编辑器与场景工作流

- **成果**：菜单 **Tools → Setup Rigged Hand Prefab** 一键从 FBX 生成左手 Prefab 并布置场景默认项；`HandSceneSetup` 使主摄像机跟随手部包围盒。
- **对应实现**：`Assets/Editor/RiggedHandPrefabSetup.cs`、`HandSceneSetup.cs`。

### 5. 控制台（ControlPanel）与物理 UI 原型

- **成果**：在 `SampleScene` 中可放置 **ControlPanel**（抬高到桌面工作高度，y≈0.72，桌面顶面世界 y≈0.74，更贴近真实操作姿势）：灰色 0.7 m × 0.04 m × 0.45 m 桌板、红/蓝两枚按压按钮（带透明触发体）、可抓取并按手位移调节的圆柱旋钮（含 `Visual` 子物体——已自动挂上默认配置的 `RotaryKnob`——与 `GrabZone` 触发球，本轮新增的 `GrabbableKnobAdapter` 由独立菜单挂到 `GrabZone`）、两盏由 `LampController` 驱动的灯泡（点光源 + 自发光材质）。菜单 **Tools → Setup Control Panel** 生成层级与材质，并将 **Button_01/02 → Lamp_01/02.Toggle** 持久化绑定（便于答辩一键还原）；菜单 **Tools → Setup Knob Interaction** 接管旋钮抓取链路（详见 §三.7）。
- **对应实现**：`Assets/Editor/ControlPanelSetup.cs`、`Assets/Editor/KnobInteractionSetup.cs`；`PressableButton.cs`、`ButtonTriggerZone.cs`、`LampController.cs`；材质目录 `Assets/Materials/ControlPanel/`。
- **与手套的关系**：控制台物体 **不需要** 挂 `GloveDataReceiver`（见下文附录）。手部交互依赖 **`FingerTip` Tag** + **`ButtonTriggerZone`** / 旋钮 `GrabZone` 上的 `TouchableObject` 抓取；`GloveDataReceiver` 仅负责把弯曲数据交给 **`DataGloveHandDriver`**（与 **`HandInteractionRig`** 捏合逻辑）。旋钮旋转由本轮新增的位移驱动链路实现（详见下文 §三.7），**不必**把每个控件都连到 `GloveDataReceiver`。

### 6. 触觉硬件侧验证

- **成果**：不依赖 Unity 即可对 HID 板发 **0xAA**（库振动）与 **0xBB**（遍历 DRV2605 效果 ID）命令，用于双路马达快速冒烟测试。当前是 §三.8 在线闭环之外的备用工具——开发期与板子第一次通电时仍优先用它做端到端验证，避免把"板子坏了"误判成"Unity/桥代码错了"。
- **对应实现**：`test_hid.py`、`test_vibrate.py`。
- **与在线闭环的互斥**：HID 板同一时刻只能被一个进程占用，所以游戏运行（`HapticBridge.py` 在跑）时不能并行运行这两个脚本，反之亦然。

### 7. 旋钮交互与亮度控制系统（持续迭代到 v12）

- **成果**：场景中的旋钮可被指尖捏取，抓取期间执行 **snap-to-grab**——虚拟手"焊"在触碰瞬间的世界位置上，但 IMU 仍持续输出 raw 目标位置；旋钮按"**每帧位移 → 速度阈值过滤 → 累加器**"（v12 速率累加，默认 `degreesPerMeter=3000` ≈ 30°/cm，约 9 cm 行程覆盖 0–270° 全量程）线性驱动角度；归一化输出经 `MasterBrightnessController` 同步驱动多盏 `LampController`，可直接演示"扭旋钮 → 一组灯同步亮度变化"。释放时把 IMU 积分器**重锚到旋钮位置**，手"原地不动"而不是被弹回胸前；底层 pinch 检测的快速抖动由 0.2 s 的去抖窗口吃掉。整套链路**不依赖 XR Interaction Toolkit**，由编辑器菜单 **Tools → Setup Knob Interaction** 一键搭建（含旧组件清理、引用接线、`onValueChanged` 持久绑定）；v11 起还提供 `useDirectFingerBendGrab` 备用路径，可在没接触发器的最小化场景里直接用手指弯曲度抓取旋钮，便于诊断。
- **关键实现 / 设计要点**：
  - **场景层级（v7 起）**：`Knob_01 / { Visual (RotaryKnob), GrabZone (TouchableObject + Rigidbody + GrabbableKnobAdapter) }`。`Knob_01` 根上**不再**挂 `TouchableObject`——旧版结构会和 `GrabZone` 链路同时触发，造成"旋钮被整体抓走、灯不亮"。`KnobInteractionSetup` 会自动清理残留组件。
  - `DataGloveHandDriver.cs`：
    - 加 **`[DefaultExecutionOrder(-200)]`**，确保下游脚本（RotaryKnob 等）读取手腕位置永远是当前帧最新值，并杜绝其他 Update 晚于位置 lock 写入而把锁顶掉的帧序竞争。
    - **`lockWristRotation`**：跳过当前帧 `transform.localRotation` 写入，其他链路全部正常跑；
    - **`lockWristPosition` + `lockedWristPositionTarget`**：每帧把 `transform.localPosition` 硬 snap 到目标点（visual snap-to-grab），但 IMU 内部 `_currentWristOffset / _wristVelocity` 继续积分；
    - **`RawWristTargetPosition`**（只读）：本帧 IMU 计算出的、未经平滑、未受 lock 影响的世界目标位置——`RotaryKnob` 抓取期间靠它继续感知真实手位移；
    - **`RebaseWristAnchorToCurrent()`**：把 IMU 积分器的"中性原点"重新锚定到当前 `localPosition`，并清空 offset / velocity / 加速度滤波——这是松手时不"传送回胸前"的关键。
  - `RotaryKnob.cs`（**v12 速率累加 + 速度阈值**）：
    - 抓取瞬间记录"上一帧手位置"和"基准角度"（不是"绝对手位置"）；每帧 `frameDelta = currentHandPos - _lastHandPos`，沿 `displacementAxis` 取分量得到 `axisDelta`。
    - **关键的 `driftVelocityThreshold`（默认 0.08 m/s）**：把 `axisDelta / dt` 当成速度，**低于阈值的帧直接丢弃不累加**——这就是治"旋钮转完灯亮一下又自己暗下去"的根因。`DataGloveHandDriver` 的位置链路有弹簧回中（`wristPositionSpring` + `positionRecenteringSpeed`），用户停手后 `_currentWristOffset` 会被慢慢拉回 0，老版本"绝对位移→角度"模式会把这个慢漂移误识别成"用户在反向拧旋钮"。
    - 累加进 `_accumulatedAxisDisplacement` 后乘 `degreesPerMeter × signFlip` 得角度，钳制到 `[minAngle, maxAngle]`，并**回写累加器**（钳制后避免"积负债"——不然推到上限后继续推，反向时要先把负债转完才能掉头）。
    - **`positionSource` + `useRawWristPosition`**：开启时优先读 `DataGloveHandDriver.RawWristTargetPosition` 而非 `handTransform.position`——这是 snap-to-grab 期间旋钮还能转动的前提（visual handTransform 已被 lock 冻结）。
    - 事件 `onValueChanged(float 0~1)`、`onStepClicked`（基于 `angleStep` 的步进检测，**触觉就是从这个事件接到 `HapticOutput.PlayKnobStep` 的**）。
    - `logAxisDiagnostics` 打开后每 ~0.3 s 打印 `sum|Δ|` / `vel` / `accum` / `angle`，是定位"选错 displacementAxis"或"阈值太严吃掉真输入"的快速工具。
  - `GrabbableKnobAdapter.cs`：
    - `OnEnable` 缓存原 parent / localPos / localRot（必须在事件订阅前抓快照——`TouchableObject.Grab()` 在 `onGrabbed.Invoke()` 之前就 `SetParent(handAttach)` 了，handler 里读到的 parent 已是手）；
    - `Update` 持续把自身 transform 钉回 `_originalParent` / pose，抵消 reparent；
    - 抓取期间根据 `lockWristDuringGrab` / `snapHandToKnob` / **`freezeAtCurrentHandPosition`**（v10 默认开启）同时打开两把锁，把虚拟手"焊"在触碰瞬间的当前位置而不是 snap 到旋钮中心；
    - **`minLockHoldDuration`**（v8，默认 0.2 s）：`HandleRelease` 不立刻解锁，而是把"应解锁时间点"记下来由 `Update` 到点执行；期间任何 `HandleGrab` 都会清掉 deadline——专治 pinch 阈值附近的 grab/release 抖动；
    - **`rebaseHandAnchorOnRelease`**（v9，默认开启）：解锁前先调 `RebaseWristAnchorToCurrent()`，让 IMU 弹簧把"新原点"记成旋钮位置；
    - 同时还做了 v8/v9 的 `GrabZone` 自身归位：`TouchableObject.Release()` 会把它 `SetParent(null)` + `kinematic = false`，必须立刻 reparent 回 `Knob_01` 并恢复 kinematic，否则下一帧 `GrabZone` 会从面板上掉下去；
    - `OnDisable` 兜底复位两把锁与 `RotaryKnob` 状态，并在解锁前同样做 rebase；
    - 默认开启 `logGrabReleaseDiagnostics`，每次抓/放在 Console 单行打印关键引用与锁状态（哪一项是 NULL 就说明对应配线缺失），用于快速定位"灯不亮 / 锁不上"。
  - `MasterBrightnessController.cs`：无状态 fan-out，`SetGlobalBrightness(float)` 同步分发到 `lamps[]` 每个 `LampController`，并在 value > 0 时自动 `SetOn(true)` 隐式点灯；**`enableKeyboardTest`** 默认开启，按 0–9 / `=` 直接设置亮度，独立于旋钮的 fallback 通道，用于诊断"亮度链路本身"是否畅通——是定位"键盘能改但旋钮不行 = 旋钮一侧问题"还是"键盘也改不了 = master/lamp 配置问题"的关键。
  - `HandInteractionRig.cs`：`enableAnySingleFingerBentGrab` / `enableInverseBentGrab` 两个宽松兜底默认**关闭**（之前 `enableInverseBentGrab` 默认 ON 导致"手张开五指≈0 = 一直在抓"，触碰旋钮立即被吸附）；阈值字段也补充了"仅在对应开关 ON 时生效"的提示。
  - **v11 直接抓取（`GrabbableKnobAdapter.useDirectFingerBendGrab`）**：在 `GrabbableKnobAdapter` 上新增一条与 `TouchableObject` / `HandInteractionRig` 完全无关的备用抓取路径——直接读 `GloveDataReceiver.FingerValues`，按 `fingerBendMode`（`AverageOfAll` / `ThumbAndIndexBoth` / `AnyFinger` / `IndexOnly`）聚合成 0–1 标量，越过 `grabBendThreshold`（默认 0.5）抓取、跌破 `releaseBendThreshold`（默认 0.25）释放。配合 `DataGloveHandDriver.enableKeyboardFistOverride` 的 Space 全握拳，**不需要任何触发器、不需要 `HandInteractionRig` 配 pinch 阈值**就能验证"手指弯曲 → 抓住 → 转动 → 灯亮"链路是否通——这是把"灯不亮"问题定位到"触发器配错"还是"旋钮/灯链路本身有问题"的关键工具。
  - `Editor/KnobInteractionSetup.cs`（新）：菜单 **Tools → Setup Knob Interaction**，做 §三.5 提到的所有接线工作，走 `Undo` 系统、Ctrl+Z 一步回退；可重复执行（先清旧 persistent listener 再重挂，不叠加）。
  - `Editor/ControlPanelSetup.cs` 改版：把 `ControlPanel` 抬到 y=0.72，桌板尺寸放大到 0.7 m × 0.04 m × 0.45 m，按钮/旋钮/灯重新布局；`Knob_01/Visual` 在生成时**自动挂 `RotaryKnob`**，省掉手动 `AddComponent`。
- **对应实现**：`Assets/Scripts/RotaryKnob.cs`、`GrabbableKnobAdapter.cs`、`MasterBrightnessController.cs`、`DataGloveHandDriver.cs`、`HandInteractionRig.cs`、`TouchableObject.cs`、`Assets/Editor/KnobInteractionSetup.cs`、`Assets/Editor/ControlPanelSetup.cs`。挂法详见根目录 `README.md` §6.5。

### 8. 触觉反馈输出系统（本轮新增）

## 附录：GloveDataReceiver 是什么？控制台 Button / Knob 要不要绑它？

**`GloveDataReceiver`（`Assets/Scripts/GloveDataReceiver.cs`）** 是 **手套数据入口**：从 **UDP**（`SensorBridge.py` 转发的串口数据）或 **键盘模拟**（无硬件调试）读取五指弯曲，解析为 **长度 5、0～1 归一化的 `FingerValues`**，供 **`DataGloveHandDriver`** 旋转手指骨骼；若场景里存在该组件，**`HandInteractionRig`** 也会在运行时自动找到它，用于 **捏合阈值 / 握拳抓取** 等基于弯曲的判断。**v11 起**，`GrabbableKnobAdapter` 在 `useDirectFingerBendGrab=true` 时也会直接订阅它的 `FingerValues`——这是仅有的两个会主动消费 `GloveDataReceiver` 数据的运行时组件。

**不需要** 在 `ControlPanel` 下的 **Button、Knob** 上逐个绑定 `GloveDataReceiver`。推荐链路是：

1. 手套数据 → `GloveDataReceiver` → 骨骼弯曲；手腕 BLE → `DataGloveHandDriver` → 手整体位姿。  
2. 运行时 **`HandInteractionRig`** 在指尖骨骼下生成带 **`FingerTip` Tag** 的球形 Trigger。  
3. **按钮**：子物体 **`TriggerZone`** 上的 **`ButtonTriggerZone`** 检测 Tag，调用 **`PressableButton.OnFingerEnter/Exit`**，即可驱动下沉与 `UnityEvent`；若同时跑了 **Tools → Setup Haptics**，按下还会触发 `HapticOutput.PlayButtonClick` → UDP → DRV2605 *Strong Click*。
4. **旋钮**：现已实现 snap-to-grab 位移驱动旋钮链路 — 期望层级是 `Knob_01 / { Visual (RotaryKnob), GrabZone (Rigidbody Kinematic + 非 Trigger Collider + TouchableObject + GrabbableKnobAdapter) }`，`Knob_01` 根上**不挂** `TouchableObject`/Adapter（v7 起）。绝大多数情况直接跑菜单 **Tools → Setup Knob Interaction** 即可一键配齐所有引用，并把 `RotaryKnob.onValueChanged` 持久绑定到 `MasterBrightnessController.SetGlobalBrightness` 同步控制多盏灯；同样，**Tools → Setup Haptics** 会把 `RotaryKnob.onStepClicked` 持久绑定到 `HapticOutput.PlayKnobStep` 让旋钮每跨一格咔嗒一下。整套流程依赖 `HandInteractionRig` 的捏合判定来触发抓取（v11 之后还可走 `useDirectFingerBendGrab` 直接抓取路径），**仍不必** 在旋钮上引用 `GloveDataReceiver`（除非你启用了 v11 直接抓取，那时 adapter 会在 `OnEnable` 自动 `FindFirstObjectByType` 找到它）。

## 六、与「完全成品」之间的差距（诚实边界，便于答辩口径）

- **XR**：当前为桌面/非 XR 演示形态，未集成 XR Interaction Toolkit。  
- **触觉闭环**：游戏内 → DRV2605 双路 LRA 的在线闭环已经跑通（按钮 *Strong Click*、旋钮步进 *Soft Click*、可选抓握 *Strong Buzz*）；剩余非项是"按物体材质/碰撞速度做更细粒度的反馈选取"以及"基于 RTP 模式做连续力度调制"，这些需要为每类交互再写一套效果库映射，后续可在 `HapticOutput` 上扩展 `Play*` 方法 + `HapticBridge.py` 加 0xCC 类型报文支持。  
- **Arduino + DRV2605**：README 列为可选替代路径；当前主路径直接走 USB HID 板，不需要 Arduino。如改用 Arduino 串口固件，可复用 `HapticBridge.py` 的 UDP 协议层、把 HID 写入替换为串口写入，Unity 端无需改动。
- **WT9011 位移边界**：当前方案强调**近场、短距离**“相对位移可演示、转动假位移更可控”，**不适合**当作远距离自由平移或绝对 3D 定位；除 `positionAxisWeight`、`gravityEstimateFilterSpeed`、`positionFreezeDuration`、`wristPositionSpring` 外，还可通过 `wristMaxOffset`、`usePlanarPositionOnly`、`positionRecenteringSpeed` 与 `wristVelocityDamping` 等结合佩戴方式与演示动作继续实测调优。  
- **旋钮链路边界**：
  - 灵敏度：`RotaryKnob` 用「**手部 raw 位移 × degrees-per-meter**」线性映射；抓取期间读 `RawWristTargetPosition`（不是 visual handTransform），所以受 `DataGloveHandDriver` 的 `wristMaxOffset`（默认 0.18 m）与 `positionRecenteringSpeed`（弹簧回中）影响 — 单次抓取行程 < 1 s 内不会受限；长时间慢速精调时回中会"吃掉"位移导致旋钮慢慢漂回，需要相应调高 `degreesPerMeter` 让动作更快完成。
  - Snap-to-grab：`lockWristPosition` 是**硬 snap**（无平滑），抓取瞬间手会立刻冻结到 `freezeAtCurrentHandPosition` 决定的目标点；松手由 `rebaseHandAnchorOnRelease` 把"原点"切到当前位置后再解锁，不会 snap 回去。`lockWristRotation` 仅冻结手腕旋转的写入（IMU 平滑仍在内部跑），释放后按 `wristSmoothSpeed`（默认 3）平滑回到实时姿态。
  - 抖动 vs 延迟：`minLockHoldDuration` 是 grab/release 去抖的核心折中——值越大越不抖、但松手感觉延迟越大；典型 0.15–0.30 s。
  - 配线脆弱点：`RotaryKnob.positionSource` / `useRawWristPosition` 与 `GrabbableKnobAdapter.handDriver` 任一缺失，整套链路就会"看起来抓住了但旋钮不动 / 灯不亮"。`Tools → Setup Knob Interaction` 与 `MasterBrightnessController` 的键盘旁路（0–9 / `=`）就是为了让这种问题在 5 秒内能定位到具体一项。
- **资源占用**：`BlueScanner` / `BlueConnector` 与 `WitMotionConnector` 并存时注意 BLE 资源，README 已提示。
